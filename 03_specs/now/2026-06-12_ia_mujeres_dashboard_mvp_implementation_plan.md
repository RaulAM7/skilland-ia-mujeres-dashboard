# IA Mujeres Dashboard MVP Implementation Plan

## Executive Summary
- Build a read-only internal dashboard for the SkilLand IA Mujeres funnel.
- The dashboard is a deployable Vite/React/Vercel app that lives at repository root alongside the agent scaffold folders.
- The dashboard must show funnel health, operational next actions, tasks, alerts and CRM-derived metrics.
- The CRM/Twenty workspace remains the source of truth.
- The dashboard owns the snapshot: server-side API queries CRM/Twenty, normalizes data, builds a compact UI contract and returns it to the frontend.
- MVP is mock-first so local UI/API development can start before real CRM credentials and runtime schema are available.
- CRM fields listed from source docs are source-confirmed but runtime-unverified until checked against the deployed API.
- No write-back, email sending, draft creation, destructive action or CRM mutation is allowed in MVP.
- No secrets can be committed and no CRM token can be exposed to browser code.

## Product Goal
- Provide a command center for the IA Mujeres institutional/commercial funnel.
- Let the team answer quickly: how the funnel is going, what changed, what is blocked, what tasks are due and what to do today.
- Convert CRM/Twenty data into an executive-operational view without duplicating the CRM.
- Keep the first product increment useful with mock data and ready for real CRM connection when credentials/schema are confirmed.

## Non-Goals
- Not a CRM replacement.
- Not an email sender, draft creator, GWS runner or batch execution tool.
- Not a scraping, enrichment or contact-modification tool.
- Not a write-back interface in MVP.
- Not a BI warehouse or historical analytics platform.
- Not a static report generated by the CRM repo.
- Not a public dashboard.

## Architecture

Canonical flow:

```text
Browser
  -> Dashboard React / shadcn-admin style UI
  -> GET /api/ia-mujeres/snapshot
  -> Vercel Function
  -> IaMujeresCrmRepository
  -> CRM/Twenty API or mock repository
  -> buildIaMujeresSnapshot()
  -> JSON snapshot
  -> UI
```

- Frontend: React, Vite, TypeScript, Tailwind, shadcn/ui, shadcn-admin style/layout, TanStack Router, TanStack Query, Recharts.
- API: Vercel Functions under root `api/`.
- Server code: root `server/` modules for CRM access and snapshot building.
- Data mode: `DASHBOARD_DATA_MODE=mock` by default; `crm` only when runtime CRM config exists.
- Snapshot builder must not know whether data came from mock or Twenty.
- UI must only consume `/api/ia-mujeres/snapshot`; never CRM directly.

## UI Pages

### Overview
- Purpose: answer "how is the funnel now?" and "what should the team do today?"
- Components: KPI cards, funnel summary, alerts panel, next actions, tasks due, manual review list, snapshot status.
- Data: `totals`, `kpiCards`, `alerts`, `tasks`, `warnings`, top `opportunities`.
- MVP: required.

### Funnel
- Purpose: inspect opportunity distribution by stage and search/filter entities.
- Components: funnel chart, stage table, opportunities table, stage/entity filters.
- Data: `funnelStages`, `opportunities`, `warnings`.
- MVP: required with mock data.

### Operation / Tasks
- Purpose: operational work queue for due tasks, follow-ups and manual review.
- Components: tasks table, follow-up list, overdue tasks, manual review list.
- Data: `tasks`, `totals.openTasks`, `totals.overdueTasks`, `totals.dueTodayTasks`, `totals.followupsPending`.
- MVP: required.

### Debug / Settings
- Purpose: show snapshot health and safe technical metadata.
- Components: snapshot status badge, source summary, warnings, error panel, safe raw snapshot in local/dev only.
- Data: `source`, `generatedAt`, `status`, `warnings`.
- MVP: required for implementation/debug, no secrets.

### Future Pages
- Batches: phase 2 unless batch fields are available in CRM.
- Organizations: phase 2 if entity classification becomes reliable.
- Historical trends: phase 2+ with persistent snapshot storage.

## Components
- `KpiCard`: compact metric card with helper/trend.
- `FunnelStageChart`: chart from `funnelStages`.
- `FunnelStageTable`: stage count/percentage/order table.
- `OpportunitiesTable`: filtered opportunities list.
- `TasksTable`: open/due/overdue task table.
- `AlertsPanel`: warning and critical operational items.
- `NextActionsPanel`: prioritized "que toca hoy" list.
- `ManualReviewList`: opportunities requiring human action.
- `SnapshotStatusBadge`: `ok`, `stale`, `partial`, `error`.
- `DebugSnapshotPanel`: safe local/dev diagnostics only.

## Snapshot Contract
- Canonical type is defined in `04_outputs/data_contract/2026-06-12_dashboard_snapshot_contract.md`.
- Required top-level keys: `schemaVersion`, `generatedAt`, `status`, `source`, `totals`, `funnelStages`, `kpiCards`, `alerts`, `tasks`, `opportunities`, `warnings`.
- Optional top-level keys: `batches`, `technicalEmailOutcomes`.
- Status values: `ok`, `stale`, `partial`, `error`.
- Source values: `mock`, `twenty`, `custom`.
- Unknown stages must map to `UNKNOWN_STAGE`, render safely and emit `UNKNOWN_STAGE` warnings.
- Empty state must return valid arrays and zero totals, not break UI.
- Error state must not include secrets, full raw CRM payloads or email bodies.

## API Endpoints

### `GET /api/ia-mujeres/snapshot`
- Purpose: main UI data endpoint.
- Input: optional query params may be added later for debug or cache bypass; none required for MVP.
- Output: `IaMujeresDashboardSnapshot`.
- Auth: dashboard-level protection for real deployment; no CRM secret in browser.
- MVP behavior:
  - `DASHBOARD_DATA_MODE=mock`: load mock repository and return valid snapshot.
  - `DASHBOARD_DATA_MODE=crm`: call `TwentyIaMujeresRepository`; if env/schema missing, return explicit `error` or `partial`.
- Errors: handle CRM unavailable, auth failure, timeout, malformed data, unknown stage, empty data.

### `GET /api/ia-mujeres/refresh`
- Purpose: refresh hook reserved for Vercel Cron.
- Auth: must require `CRON_SECRET`.
- MVP: may rebuild mock snapshot or return "not configured" until persistent cache exists.

### `POST /api/ia-mujeres/refresh`
- Purpose: manual admin refresh.
- Auth: must require `DASHBOARD_REFRESH_SECRET`.
- MVP: optional implementation after snapshot endpoint; never public.

### `POST /api/webhooks/twenty`
- Purpose: future cache invalidation/regeneration from CRM events.
- Auth: future webhook secret/signature.
- MVP: out of scope.

## CRM/Twenty Adapter

Use an adapter so UI and snapshot builder do not depend on Twenty-specific response shapes.

```ts
type IaMujeresCrmRepository = {
  listCampaignOpportunities(params: {
    campaignKey: string
  }): Promise<RawIaMujeresOpportunity[]>

  listCampaignTasks(params: {
    campaignKey: string
  }): Promise<RawIaMujeresTask[]>

  listCampaignCompanies?(params: {

  }): Promise<RawIaMujeresCompany[]>

  listCampaignPeople?(params: {
    campaignKey: string
  }): Promise<RawIaMujeresPerson[]>

  getHealth?(): Promise<CrmHealthStatus>
}
```

## Security Model
- Read-only MVP.
- Server-only CRM access.
- No email bodies in snapshots.
- No secrets in repo, logs, client code or debug panels.
- Dashboard must be protected before real CRM data is exposed.
- Snapshot should minimize personal data; organization/stage/next action is preferred over person-level data.
- Refresh endpoints are protected by different secrets for cron and manual admin.

## Cache Strategy
- MVP: no persistent cache required.
- MVP may use short HTTP cache headers or in-function memoization only if simple.
- If CRM fails and no cache exists, return a safe `error` snapshot.
- Phase 2: Vercel KV/Redis/Blob, stale-while-revalidate, last successful snapshot, webhook invalidation and cron fallback.
